# Coding Rules

実装時に従うコーディング規則（curated doc）。**コードを書き始める前にこのファイルを読む。**

このプロジェクトの方針: **機械的に強制できる規則（整形・スタイル・import 順・言語固有の型ルール）は lint / formatter / typechecker に委譲し、この doc には「判断を要する規則」だけを置く。** スタイルを prose で細かく書いても破られ得るが、ツールは破られない。だから prose は薄く、ツールに厚く寄せる。

## 段階で運用が変わる

`04-ARCHITECTURE.md` のスタック表と同じく、段階で運用が変わる。

- **探索 / MVP 期（スタック未 pin、lint 等が未整備）**: 下記の prose ルールを手動で守る。
- **pin 後（lint / formatter / typecheck 整備済み）**: スタイル系はツールに任せ、この doc は判断系ルールだけを保持する。ツールの設定（eslint / prettier / ruff / tsconfig 等）が prose と矛盾したら、**ツール設定を正**とする。

## Claude のデフォルトに任せる（ここで再掲しない）

以下は Claude Code が既定で守るので、この doc では繰り返さない（薄く保つため）。逸脱が必要なときだけ個別に明記する。

- コメントは原則書かない（WHY が非自明なときだけ。WHAT・タスク/呼び出し元への言及は書かない）
- タスクが要求する以上の抽象化・機能・リファクタを足さない（YAGNI、早すぎる共通化を避ける）
- 起こり得ないケースのエラーハンドリング／バリデーションを足さない（検証は境界だけ）
- 新規ファイルより既存ファイルの編集を優先
- 後方互換ハック（未使用 `_var` リネーム、`// removed` コメント等）を避ける
- セキュリティ脆弱性（command injection / XSS / SQLi 等）を作らない、secret をコードに書かない

## このプロジェクトで明示する規則

判断を要する／デフォルトに無い／数値で示したい規則だけ。**数値はすべて「目安」。** 超えること自体は禁止ではなく「分割・抽出を検討する閾値」。理由を理解して、エッジケースは自分で判断する。

### 大きさ

- 関数は目安 **40 行**。超えるなら処理を抽出する。理由: 1 画面で読めて意図を追える単位に保つ。
- ファイルは目安 **300 行**（docs と同じ閾値）。超えたら責務で分割を検討。
- 引数は目安 **4 個**。超えるならオブジェクト / struct にまとめる。

### 構造・可読性

- 早期 return でハッピーパスを浅く保つ。ネストは目安 **3 段**まで。
- マジックナンバー・マジック文字列は名前付き定数にする（意味を名前で示す）。
- 賢く短いコードより、素直で追えるコードを選ぶ。

### 命名

- その言語の慣習に従う（lint が強制するならツールが正）。
- 真偽値は `is` / `has` / `should` などの接頭辞で意図を示す。
- 広く知られた略語（id, url, db 等）以外は略さない。

### エラーハンドリング

- 例外を握りつぶさない（空 catch 禁止）。やむを得ず握るなら理由をコメントで残す。
- 失敗は早く・明示的に（fail fast）。握って続行し、壊れた状態を伝播させない。
- エラーメッセージに文脈（何をしようとして何が起きたか）を含める。

### テスト

- 実装詳細ではなく**振る舞い**をテストする（内部構造を変えてもテストが壊れないように）。
- バグ修正には再現テストを 1 つ付ける（同じ回帰を二度踏まない）。
- 具体的なテスト方針・ツールは `06-VALIDATION.md` に従う。

### 依存・スタック

- 新規ライブラリ追加は `04-ARCHITECTURE.md` のスタック表のルールに従う（pin 後は ADR 先行）。標準ライブラリで足りるなら依存を足さない。
- レイヤ依存方向は `04-ARCHITECTURE.md` の「レイヤ方針」に従う（逆流させない）。

### コミット粒度

- 1 コミット = 1 関心事。バグ修正に無関係なリファクタを混ぜない（`/checkpoint` の 1 作業単位と揃える）。

## 検証（lint ゲート）

- pin 後は **commit 前に「利用可能な検証（typecheck / lint / format / test）」を通す**。緑の定義と手順は `02-GUIDELINES.md`「チェックポイント」と `/checkpoint` に従う。
- 整形（インデント・クォート・import 順・行幅など）は formatter に任せ、この doc では規定しない。
- lint / formatter / typecheck を導入したら、設定ファイルへのリンクをこの節に追記する。
