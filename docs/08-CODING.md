# Coding Rules

実装の前に読む（環境の前提・制約は `09-ENVIRONMENT.md`）。整形・スタイル・import 順・言語固有の型ルールはツール（formatter / linter / typecheck）に任せる。下記は**判断を要する規則**だけ。数値はすべて「目安」＝分割・抽出を検討する閾値であって禁止ではない。

## 最初にコードを書くときにツールを入れる

新しい言語で最初にコードを書くとき、**実装より先に**その言語の標準 formatter / linter / typecheck をデフォルト構成で導入する。

- ユーザに聞かない。デフォルト構成の導入は引き返せる決定。選んだツールを progress の「決めたこと」に1行残す。
- 標準的な選択: TS/JS → ESLint + Prettier（+ `tsc --strict`） / Python → Ruff（+ pyright か mypy） / Go → `gofmt` + `go vet` / Rust → `rustfmt` + Clippy。迷ったらエコシステムの定番を採る。
- 導入したら下記「検証」節に設定ファイルへのリンクを足す。
- 使い捨ての MVP（`mvp/`）/ PoC スパイク（`poc/`）は省略してよい（その旨を progress に残す）。

以後、**commit の前にこれらを通す**。全通過が checkpoint の「緑」の条件（`02-GUIDELINES.md`「チェックポイント」）。

## Claude の既定はそのまま使う

コメント最小（WHY が非自明なときだけ）、YAGNI・早すぎる抽象化の回避、境界だけで検証、既存ファイル優先、後方互換ハック回避、secret や injection を作らない —— これらは既定どおり守る。逸脱が要るときだけ、その箇所を明記する。

## 大きさ

- 関数は **40 行**を超えたら処理を抽出する（1 画面で意図を追える単位に保つ）。
- ファイルは **300 行**を超えたら責務で分割する。
- 引数が **4 個**を超えたらオブジェクト / struct にまとめる。

## 構造・可読性

- 早期 return でガードし、ハッピーパスを浅く保つ。ネストは **3 段**まで。
- マジックナンバー・マジック文字列は名前付き定数にする。
- 賢く短いコードより、素直で追えるコードを書く。

## 命名

- その言語の慣習に従う（lint が強制するならツールに従う）。
- 真偽値は `is` / `has` / `should` などの接頭辞を付ける。
- 広く知られた略語（id, url, db 等）以外は略さない。

## エラーハンドリング

- 例外を握りつぶさない（空 catch 禁止）。やむを得ず握るなら理由をコメントで残す。
- fail fast。壊れた状態のまま続行・伝播させない。
- エラーメッセージに文脈（何をしようとして何が起きたか）を入れる。

## テスト

- 実装詳細ではなく**振る舞い**をテストする（内部構造を変えてもテストが壊れないように）。
- バグ修正には再現テストを 1 つ付ける。
- 具体的な方針・ツールは `06-VALIDATION.md` に従う。
- 置き場所と再利用は `06-VALIDATION.md`「テストの置き場所と再利用」/ `04-ARCHITECTURE.md` ディレクトリ方針（本実装は `app/`・言語の慣習に従う。MVP のは使い捨て・引き継がない）。

## 依存・レイヤ

- 新規ライブラリは `04-ARCHITECTURE.md` のスタック表のルールに従う（pin 後は ADR 先行）。標準ライブラリで足りるなら足さない。
- レイヤ依存方向は `04-ARCHITECTURE.md`「レイヤ方針」に従い、逆流させない。

## コミット粒度

- 1 コミット = 1 関心事。バグ修正に無関係なリファクタを混ぜない（`/checkpoint` の 1 作業単位と揃える）。

## 検証

- commit 前に利用可能な検証（typecheck / lint / format / test）を通す。緑の定義と手順は `02-GUIDELINES.md`「チェックポイント」と `/checkpoint`。
- 整形（インデント・クォート・import 順・行幅）は formatter に任せ、ここでは規定しない。
