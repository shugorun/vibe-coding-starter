# Roadmap

進め方と優先順位。フェーズは順に進む。**各フェーズの移行ゲートが全チェックになるまで次フェーズに進まない**（未チェックなら残タスクを提示して現フェーズに留まる）。現在地は `05-PROGRESS.md`。

**ゲート設計の原則: 各ゲートは「次フェーズを始めるのに最低限必要な決断・入力」を含む。** 「終わった証明」ではなく「次に進む準備ができた証明」。不足したまま進むと手戻りになる（例: MVP を作るには言語が要る → 言語決定は MVP の前 = P1 のゲート）。

## フェーズと移行ゲート

### P0 ボイラープレート初期化

→ P1 のゲート:

- [ ] docs / skills / hooks が配置済み（テンプレ初期状態）

### P1 アイデア選定

作るものを 1 つに決める。

→ P2 のゲート:

- [ ] 作るアイデアを `docs/ideas/<slug>.md` に記録
- [ ] `01-PRODUCT.md` にプロダクト名・一言説明・主要ユーザ・価値仮説の初版
- [ ] MVP で試す中核体験（コンテンツ）を 1 行で言える
- [ ] **MVP を最小コストで作る手段（種別・言語 / FW）を決める**（MVP を作るのに必須。**本実装の種別とは別でよい** —— 例: 本番はモバイルでも MVP はウェブで軽く試す。使い捨て前提なので pin せず progress + `mvp/README.md` に記録）

### P2 MVP（使い捨て / `mvp/`）

`mvp/` で最小フロント＋中核機能だけを素早く作り、**このアプリの中核体験（コンテンツ）が何かを確定**する。コードの綺麗さは問わないが、比較・切り戻したい実験変種（プロンプト / スキーマ / 検索設定）は切替可能に保つ。記録は `docs/mvp-design/`（スコープ・実験・切替・フィードバック・問題）。

→ P3 のゲート:

- [ ] `mvp/` で最小フロント＋中核機能が動く（手動確認 or smoke 緑）
- [ ] 中核体験が確定。実験・学びを `docs/mvp-design/` に記録し、確定した学び（活かす / 捨てる）を `docs/progress/` / `app-design` へ昇格
- [ ] 本実装(P5)は `app/` で 0 から作る前提を確認（MVP コードは引き継がない）

### P3 app-design（詳細設計）

MVP で確定したコンテンツを活かす本実装の設計を詰める。**本実装の種別・言語・スタックは 0 から決める**（MVP の技術選定は引き継がない —— コンテンツの学びだけ引き継ぐ）。

→ P4 のゲート:

- [ ] `app-design/overview.md` に確定スコープ（作る / 作らない）
- [ ] P2 を活かす機能（認証ほか）を app-design に列挙
- [ ] **本実装のアプリ種別・言語・スタック**（種別 / 言語 / FW / バックエンド / DB / 認証 / ホスティング / 監視 / テスト）を 0 から確定し `04-ARCHITECTURE.md` のスタック表を pin（`🔍 reviewed`）
- [ ] pin に伴う一方通行決定をユーザ確認済

### P4 機能仕様

本実装に向け機能ごとに spec を書き出す。

→ P5 のゲート:

- [ ] 主要機能ごとに `docs/specs/<feature>/spec.md`
- [ ] 各 spec に受け入れ条件（`06-VALIDATION.md`）
- [ ] spec が含む一方通行決定をユーザ確認済（`🔍 reviewed`）

### P5 本実装（`app/` / 0 から）

`app/` で本番コードを新規に書く。MVP のコード/ロジックは参照せず作り直す。

→ P6 のゲート:

- [ ] `app/` で主要フローが動く
- [ ] 利用可能な検証（型 / lint / unit / build / smoke のうち存在するもの）が緑
- [ ] 本実装中の非自明な判断が ADR / progress に記録済

### P6 リリース準備

完了ゲート:

- [ ] `07-RELEASE.md` のチェックリスト完了

## 優先順位ルール

1. 中核体験（コンテンツ）に近いものを先にやる。
2. 技術リスクが高いものは MVP / PoC で早く潰す。
3. 一方通行決定はフェーズ移行ゲートでユーザ確認する。
4. 個人開発で運用負荷が高いものは後回しにする。

## 更新ルール

- フェーズ移行時、ゲートを満たしたら `05-PROGRESS.md` と progress に記録する。

- 優先順位の非自明な判断は ADR に残す。
