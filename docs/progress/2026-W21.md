## 2026-05-23（追記: stop フック判定ロジックの見直し）

### やったこと

- `stop.cjs` の発火条件を「dirty かつ progress ログが存在する」→「dirty かつ **今日の progress エントリが無い**」に変更し、`stop_hook_active` ループガードを追加（app3 で先に育っていた版を逆輸入）。実行確認 OK（今日は記録済みのため正しく沈黙）。

### 決めたこと

- stop フックは today 判定方式を採用する。理由: **Stop フックは応答ごと（毎ターン）に発火**し `systemMessage` は重複抑制が無いため、「dirty のみ」だと開発中ほぼ全ターンで通知が出てノイズ化する。「今日まだ未記録」を条件にすると、checkpoint 規律を破っている間だけ鳴り、最初の checkpoint で自動的に沈黙する（信号に紐付き・コンプライ即停止）。
- session 単位の1回限り抑制は**入れない**。バックストップは守るまで促し続け、守った瞬間止まるのが正しく、1回で諦めると記録漏れの日を見逃すため。
- 「dirty のまま終了」の検知は Stop フックの責務にしない。Stop はセッション末を判別できない（毎ターン発火）ため、それは `/wrapup-project`（明示）と `/catchup-project`（dirty start 復元）が担う。

## 2026-05-25

### やったこと

- コーディング規則の curated doc `docs/08-CODING.md` を新規作成。
- 実装時に必ず読ませる導線を 5 か所に追加: `CLAUDE.md`（「コードを書くとき」節 + 入口リンク）/ `00-INDEX.md` 必読順 / `02-GUIDELINES.md` 実装ルール先頭 / `catchup-project` 実装文脈チェック / `checkpoint` 検証節の lint ゲート参照。

### 決めたこと

- コーディング規則は「機械的に強制できる規則は lint / formatter / typecheck に委譲、prose には判断を要する規則だけ」を背骨にする。理由: スタイルを prose で書いても破られ得るが、ツール設定は破られない。curated doc も薄く保てる。
- 段階運用をスタック表と揃える: 探索/MVP 期は prose を手動順守、pin 後はスタイル系をツールへ寄せ、ツール設定と prose が矛盾したらツールを正とする。
- Claude のデフォルト（コメント最小・YAGNI・境界バリデーション・secret/injection 対策等）は再掲せず、薄いポインタ列挙だけ残す。理由: 重複はノイズで全体の効きを薄める。doc の自己完結性は「再説明せずポインタで列挙」で両立。
- 数値（関数40行 / ファイル300行 / 引数4個 / ネスト3段）は全て「目安＝分割検討の閾値」とし、厳格な上限にしない。理由: 厳格化すると無意味な分割を誘発する。
- 言語固有ルール（`any` 禁止・型注釈必須・docstring 必須等）と具体ツール名は、スタック未確定のため今は書かず pin 後に lint/tsconfig へ委譲する方針だけ記載。
