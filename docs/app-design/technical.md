# Technical Design

## 記録方法

- 技術選定、構成、API、DB、認証、運用リスクを書く。ここは検討中の作業場。
- 確定した技術構成は `04-ARCHITECTURE.md` に反映する。
- 非自明な決定は ADR に分離し、このファイルからリンクする。
- 調査が必要なものは research にトピック単位で記録する。

## プラットフォーム & 言語（P1 で確定済み）

アプリ種別・言語は **P1（MVP を作る前）** で一方通行決定として確定し、`04-ARCHITECTURE.md` のスタック表と `01-PRODUCT.md`「対象プラットフォーム」に pin 済み。ここでは再決定せず確定値を確認し、残りのスタックを詰める。MVP で言語が不適と分かった場合だけ ADR で見直す。

| 項目 | 決定（P1 で確定） | メモ |
|---|---|---|
| アプリ種別 | 未設定 | Web / モバイル / デスクトップ / CLI・ツール / その他 |
| 言語 / ランタイム | 未設定 | 例: Web→TS、モバイル→Swift / Kotlin / Dart(Flutter) / TS(RN)、CLI→Go / Rust |
| 主要フレームワーク | 未設定 | 言語に応じて。MVP で使ったものを引き継ぐか見直す |

言語の formatter / linter / typecheck は最初の実装時に自動導入する（`08-CODING.md`「最初にコードを書くときにツールを入れる」）。残りのスタック（バックエンド / DB / 認証等）は下の「技術スタック候補」で詰め、P3 ゲートで pin する。
