# Technical Design

## 記録方法

- 技術選定、構成、API、DB、認証、運用リスクを書く。ここは検討中の作業場。
- 確定した技術構成は `04-ARCHITECTURE.md` に反映する。
- 非自明な決定は ADR に分離し、このファイルからリンクする。
- 調査が必要なものは research にトピック単位で記録する。

## プラットフォーム & 言語

app-design に入ったら**最初に**ここを決める。順序は **アプリ種別 → 言語 → その先のスタック**。これは**一方通行（後から変えるのが高コスト）な決定**なので、選択肢とトレードオフを提示してユーザ確認を取ってから確定する（`02-GUIDELINES.md`「レビューゲート」）。確定したら `01-PRODUCT.md`「対象プラットフォーム」にも反映する。

| 項目 | 決定 | 候補・トレードオフ | status |
|---|---|---|---|
| アプリ種別 | 未設定 | Web / モバイル / デスクトップ / CLI・ツール / その他 | todo |
| 言語 | 未設定 | 種別に応じて（例: Web→TS、モバイル→Swift / Kotlin / Dart(Flutter) / TS(RN)、CLI→Go / Rust） | todo |
| ランタイム / 主要フレームワーク | 未設定 | 言語決定後に絞る。詳細は下の「技術スタック候補」へ | todo |

言語が決まれば、その言語の formatter / linter / typecheck の導入は最初の実装時に自動で行う（`08-CODING.md`「最初にコードを書くときにツールを入れる」）。
