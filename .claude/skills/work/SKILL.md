# Work

**発動条件**: description のとおり。
**成功基準**: 各作業単位が /checkpoint まで完了し、確定した判断がすべて docs/DECISIONS.md に1行ずつ残っている。

## 0. 現在地

`docs/STATE.md` でフェーズを確認する。作業場はフェーズで変わる（P2 = mvp/、P5 = app/、技術スパイク = poc/）。フェーズの移行・後退は `docs/ROADMAP.md` のゲートと手続きに従う。

## 1. 着手前確認（固定点）

- 設計・計画が曖昧なら /grill で合意形成してから着手する。
- 一方通行決定（基準: `docs/adr/README.md`）が絡むなら、ゲートを待たずユーザ確認を先に取る。
- **ユーザがやるべきタスク**（アカウント・認証情報・実機・ユーザにしか決められない選択）を洗い出し、未解決のまま着手しない。
- 引き返せる選択は確認不要 — 判断して進める。
- 確認は一度きりではない。作業中に新しいブロッカーが出たらここに戻る。

## 2. 分解

依頼を作業単位（= コミット1個分の動く増分）に割り、単位ごとに進める。順序は中核体験に近いものを先に。技術リスクは poc/ で早めに潰し、運用負荷が高いものは後回しにする。

## 3. 作業ループ（不変条件 — 随時発火）

- **判断が確定したら、その場で `docs/DECISIONS.md` に1行追記する**（書式は同ファイルのヘッダ）。後回しにしない。
- コードを書く前に `docs/ARCHITECTURE.md`（スタック表・コード判断規則・検証方針）と `docs/ENVIRONMENT.md` を読む。
- 解決すべき問題が残ったら /to-issues で起票する。
- 環境起因の制約（コードを直しても消えない）に当たったら `docs/ENVIRONMENT.md` に1行。
- 調査結果は `docs/research/` にトピック単位で記録する。
- MVP(P2) の実験・学びは `docs/mvp.md` に追記する。

## 4. 区切り（固定点）

/checkpoint を実行する。次の単位があれば 3 に戻る。

## 厳守事項

- ユーザ側タスクを保留したまま実装を進めない。
- ゲート未チェックのままフェーズを移行しない。
- 同一問題への試行が2回失敗したらアプローチを変えるか、issue 起票してユーザに相談する。
