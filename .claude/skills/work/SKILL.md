# Work

自然文の作業依頼を受けたら、このフローで進める。`/catchup-project`（開始）と `/wrapup-project`（終了）の間の通常作業の本体。各ステップで該当規則を読むことが目的（起票漏れと前提崩れを防ぐ）。

## 0. 現在地を把握

`docs/05-PROGRESS.md` で現在フェーズを確認する。コーディング先はフェーズで変わる: **P2 は `mvp/`（使い捨て）、P5 は `app/`（本実装・0 から）。** フェーズ移行が絡むなら `docs/03-ROADMAP.md` のゲートを確認し、未チェックがあれば移行しない（残タスクを提示して現フェーズに留まる）。例: **P2(MVP) に入るには P1 ゲートのアプリ種別・言語が確定済みであること**（未確定ならまず確定する）。

## 1. 確認（着手前）

ゴール達成までに**先に潰すべき不確実性**を洗い出す。

- ブロッカーになる質問（仕様・一方通行決定）。
- **ユーザがやるべきタスク**: アカウント作成、API キー / 認証情報、課金、外部サービス設定、実機・端末準備、ユーザしか決められない選択。
- これらが未解決なら、**着手せずユーザに提示する**（保留のまま進めて失敗させない）。一方通行決定は `docs/02-GUIDELINES.md`「レビューゲート」に従い確認する。
- 引き返せる範囲の判断は確認不要。判断して進め、根拠は progress に残す。

## 2. コーディング

- 実装前に `docs/08-CODING.md`（規則）と `docs/09-ENVIRONMENT.md`（環境の罠・ツール版）を読む。
- 新しい言語の初回は formatter / linter / typecheck を入れる（`08-CODING.md`）。
- スタックは `docs/04-ARCHITECTURE.md` の表に従う（pin 後は ADR 先行、探索/MVP 期は progress 一行 + `technical.md`）。
- **詰まり・罠・既知不具合を踏んだら、その場で `docs/issues/open/` に起票する**（後回しにしない）。環境起因なら `docs/09-ENVIRONMENT.md`「既知の罠」にも 1 行残す。

## 3. docs 更新

- `docs/02-GUIDELINES.md`「作業中の記録ルール」の表に従い、該当 docs を更新する。
- **非自明な判断（技術選定・方針・一方通行）をしたら `docs/decisions/`（ADR）に起票する。** MVP 期の小さな選定は progress「決めたこと」一行でよい（2層: GUIDELINES「意思決定の記録」）。
- 実装で仕様が変わったら spec / app-design を直す。

## 4. commit

`/checkpoint` を行う（progress 1 エントリ → 検証緑 → commit。push しない）。手順は checkpoint スキル。

## 厳守事項

- 各ステップで該当規則 doc を読んでから動く（コーディング → `08-CODING` / `09-ENVIRONMENT`、docs更新 → GUIDELINES「記録ルール」）。
- **issue / ADR の起票を後回しにしない**（起票漏れがこのプロジェクトの主要な失敗）。
- ユーザ側タスクを保留したまま実装を進めない。
- フェーズ移行ゲートが未チェックなら次フェーズに進まない。
- 軽微な 1 ステップ作業（1 行修正等）でも、最低限ステップ 2（規則確認）と 4（記録 + commit）は省かない。
