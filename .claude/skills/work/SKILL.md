# Work

自然文の作業依頼を受けたら、このフローで進める。

前後に固定点（着手前の確認・区切りの commit）があり、その間は作業ループ。規則を読む・問題→issue・判断→ADR は作業中の状況に応じて行う。

```text
確認（前提条件が未解決なら着手しない）
 └ 依頼を作業単位（= コミット1個分）に分解
    └ 各単位: 作業（コード ⇄ docs、規則を読む / 問題→issue / 判断→ADR）→ /checkpoint
```

## 0. 現在地を把握

`docs/05-PROGRESS.md` で現在フェーズを確認する。コーディング先はフェーズで変わる: **P2 は `mvp/`（使い捨て）、P5 は `app/`（本実装・0 から）。** フェーズ移行が絡むなら `docs/03-ROADMAP.md` のゲートを確認し、未チェックなら移行しない。例: **P2(MVP) に入るには P1 ゲートで「MVP を作る手段（種別・言語）」が決まっていること**（最小コスト優先・本実装と別でよい）。本実装(P5)の種別・言語・スタックは P3 で 0 から決め、MVP の技術選定は引き継がない。

## 1. 確認（着手前のゲート・固定点）

ゴール達成までに**先に潰すべき不確実性**を洗い出す。前提条件が未解決のまま着手しない（保留して進めると失敗する）。

- ブロッカーになる質問（仕様・一方通行決定）。一方通行決定は `docs/02-GUIDELINES.md`「レビューゲート」に従い確認。
- **ユーザがやるべきタスク**: 外部サービス連携、実機・端末準備、、Claudeが編集できないファイルの変更、ユーザしか決められない選択。
- 引き返せる選択は確認不要 —— 判断して進め、根拠は progress に残す。
- 確認は一度きりではない。作業中に新しいブロッカーが出たら、その時点でここに戻る。

## 2. 分解

依頼を**作業単位（= コミット1個分の動く増分）**に割る。大きい依頼ほど複数単位になる。各単位を順に下の「作業ループ」で進め、単位ごとに checkpoint する（1 単位 = 1 commit = 1 progress エントリ）。

## 3. 作業ループ（各単位）

作業単位を仕上げる。中身はコード / docs / 調査 / 混在のどれでもよく、行き来してよい。各単位で守る不変条件:

- **触る前に規則を読む**: コードを書く前に `docs/08-CODING.md`（コーディング規則）と `docs/09-ENVIRONMENT.md`（環境の事実・制約）。docs を編集する前に `docs/02-GUIDELINES.md`（記録ルール）。
- **スタックに従う（従う先は作業場で変わる）**: 本実装(`app/`)は `docs/04-ARCHITECTURE.md` の pin 済みスタック表（逸脱・追加は ADR 先行）。MVP(`mvp/`)は `docs/mvp-design/technical.md` の使い捨て構成（ADR 不要）。
- **解決すべき問題に当たったら → その場で `docs/issues/open/` に issue 起票**（定義は `issues/_index.md`）。後回しにしない。
- **コードを直しても消えない環境起因の制約に当たったら → `docs/09-ENVIRONMENT.md` に 1 行。**
- **非自明な判断をしたら → `docs/decisions/` に ADR 起票**（MVP 期の小さな選定は progress「決めたこと」一行で可。2層: `02-GUIDELINES.md`「意思決定の記録」）。
- 実装で仕様が変わったら spec / app-design を直す。
- **P2(MVP) のとき**: 実験のバージョン（プロンプト / スキーマ / 検索設定）を切替可能に保つ。記録は `docs/mvp-design/`（実験→`experiments.md` / バージョン→`switches.md` / フィードバック→`feedback.md` / 困りごと→`problems.md`）。

## 4. checkpoint（単位の区切り・固定点）

`/checkpoint` を行う（progress 1 エントリ → 検証緑 → commit。push しない）。手順は checkpoint スキル。次の単位があれば「3. 作業ループ」に戻る。

## 厳守事項

- 一直線に踏まない。**着手前の確認**と**区切りの commit** が固定点、規則読み・問題→issue・判断→ADR は随時発火する不変条件。
- **issue / ADR の起票を後回しにしない**（起票漏れがこのプロジェクトの主要な失敗）。
- ユーザ側タスクを保留したまま実装を進めない。
- フェーズ移行ゲートが未チェックなら次フェーズに進まない。
- 軽微な 1 単位（1 行修正等）でも、規則確認と「記録 + commit」は省かない。
