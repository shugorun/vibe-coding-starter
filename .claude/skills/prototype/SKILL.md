# Prototype

出典: [mattpocock/skills](https://github.com/mattpocock/skills) の prototype を改変（MIT — 表記は [LICENSE](../../../LICENSE)）。

**発動条件**: description のとおり。
**書き込み**: コードのみ。答え（学び）はチャットで報告し、docs への記録は各レーンの規約（mvp/ → docs/mvp.md、poc/ → docs/research/）に従って /work・/checkpoint が行う。

プロトタイプとは**問いに答えるための使い捨てコード**。問いの種類が形を決める。

## 分岐を選ぶ

どの問いに答えるのかを、依頼文・周辺コード・（不明ならユーザへの質問で）特定する:

- **「このロジック / 状態モデルで合っているか?」** → [LOGIC.md](LOGIC.md)。紙の上では正しそうな状態機械を、判断しづらいケースに通す小さな対話ターミナルアプリを poc/ に作る。
- **「これはどう見えるべきか?」** → [UI.md](UI.md)。現フェーズの作業場（P2 = mvp/、P5 = app/）の該当ページに、`?variant=` で切り替える構造的に異なる複数 UI 案をマウントする。

2つの分岐は全く異なる成果物を生む — ここを間違えるとプロトタイプ全体が無駄になる。問いが本当に曖昧でユーザにも聞けないときは、周辺コードに近い方（バックエンドモジュール → ロジック、ページ / コンポーネント → UI）を選び、仮定を成果物の先頭に明記する。

## 両分岐に共通のルール

1. **初日から使い捨てであり、そうと分かるように。** 使う場所の近くに置きつつ、名前で「プロトタイプであって本番ではない」と一目で分かるようにする。
2. **1コマンドで起動できる。** ユーザが何も考えずに開始できること。
3. **永続化しない。** 状態はメモリ内。永続化はプロトタイプが*確かめる*対象であって、依存してよい前提ではない。
4. **磨かない。** テストなし・起動に必要な分以上のエラー処理なし・抽象化なし。速く学んで捨てるのが目的。
5. **状態を表に出す。** 操作のたび（ロジック）・案の切替のたび（UI）に関連する状態すべてを表示する。
6. **終わったら消すか吸収する。** 問いに答えたら、削除するか、検証済みの決定だけを本コードに反映する。リポジトリに腐らせない。

## 終わったら

プロトタイプから残す価値があるのは**答えだけ**。問いと答えをセットでユーザに報告し（記録先は冒頭の「書き込み」のとおり）、プロトタイプ本体は削除する。
