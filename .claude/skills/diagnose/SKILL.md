# Diagnose

出典: [mattpocock/skills](https://github.com/mattpocock/skills) の diagnose を改変（MIT — 表記は [LICENSE](../../../LICENSE)）。

**発動条件**: description のとおり。フェーズの省略は明示的に正当化できるときだけ。
**書き込み**: コードと、docs/issues/ の該当 issue の**更新**のみ（試した対処・解決・status。書式は [issues/README.md](../../../docs/issues/README.md)）。対象が未起票なら、先に /to-issues で「問題」として起票してから始める。

探索時は [PRODUCT.md](../../../docs/PRODUCT.md)「用語」節で関連モジュールの心的モデルを作り、触る領域の [adr/](../../../docs/adr/README.md) に従う。

## Phase 1 — フィードバックループを作る

**ここがこのスキルの本体。残りは機械的な消費にすぎない。** 速く・決定的で・エージェントが自走できる pass/fail シグナルがあれば、バグは見つかる。なければ、コードを睨んでも何も出ない。

構築手段10種（10種は [feedback-loops.md](feedback-loops.md)）・ループ自体の改善・非決定的バグの扱いは同ファイル。HITL（人の操作が必要な再現）は [scripts/hitl-loop.template.cjs](scripts/hitl-loop.template.cjs) をコピーして編集する（Node 製・全 OS 共通）。

**どうしてもループが作れないときは止まって明言する。** 試したことを列挙し、ユーザに (a) 再現環境へのアクセス (b) 採取物（HAR・ログ・録画）(c) 一時的な計測コード追加の許可、のいずれかを求める。ループなしで Phase 3 に進まない。

## Phase 2 — 再現

ループを回してバグの発生を見る。確認: ユーザが報告した失敗モードそのものか（近くの別バグではないか）/ 複数回再現するか / 正確な症状（エラー文・誤出力・時間）を捕捉したか。再現するまで先に進まない。

## Phase 3 — 仮説

検証を始める前に**ランク付けした仮説を3-5本**出す（1本だけだと最初の思いつきに固着する）。各仮説は反証可能に: 「<X> が原因なら、<Y を変えれば> 消える / <Z を変えれば> 悪化する」。予測が言えない仮説は雰囲気なので捨てるか研ぐ。

**検証前にランク付きリストをユーザに見せる**（ドメイン知識で即座に並べ替えられることが多い）。ただしブロックはしない — 返事がなければ自分のランクで進む。

## Phase 4 — 計測

各プローブは Phase 3 の特定の予測に対応させる。**一度に変えるのは1変数。** 道具の優先順: ①デバッガ / REPL ②仮説を区別する境界への狙い撃ちログ ③「全部ログして grep」は禁止。

**デバッグログには必ず一意タグ**（例: `[DEBUG-a4f2]`）を付ける — 後始末が grep 1回になる。性能劣化はログでなく、まず基準計測（タイミングハーネス・プロファイラ）→ bisect。計測してから直す。

## Phase 5 — 修正 + 回帰テスト

回帰テストは**修正の前に**書く — ただし**正しい継ぎ目**がある場合のみ。正しい継ぎ目 = 呼び出し箇所で実際に起きたバグパターンをそのまま運動させられる場所。浅すぎる継ぎ目のテストは偽の安心になる。**正しい継ぎ目が無いならそれ自体が発見** — issue に書き残す。

継ぎ目があるなら: 最小化した再現を失敗テストにする → 失敗を見る → 修正 → 通過を見る → Phase 1 のループを元の（最小化前の）シナリオで再実行。

## Phase 6 — 後始末 + 記録

完了宣言の前に必ず:

- [ ] 元の再現が再現しなくなった（Phase 1 のループを再実行）
- [ ] 回帰テストが通る（または継ぎ目が無いことを issue に記録した）
- [ ] `[DEBUG-...]` 計測コードを全削除（タグを grep）
- [ ] 使い捨てハーネスを削除
- [ ] 該当 issue を更新: 試した対処 / 解決（正しかった仮説を書く — 次の診断者が学べる）/ `status: resolved`

最後に「何がこのバグを防げたか」を自問する。構造的な対策（継ぎ目の欠如・絡んだ呼び出し元）が要るなら、修正を入れた**後で** /to-issues での起票を提案する — 始めた時より今のほうが情報がある。
