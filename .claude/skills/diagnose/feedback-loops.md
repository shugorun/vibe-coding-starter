# フィードバックループの構築手段

[SKILL.md](SKILL.md) Phase 1 の詳細。上から順に試す。

## 手段（おおよそこの順で）

1. **失敗テスト** — バグに届く継ぎ目ならどこでも（unit / integration / e2e）。
2. **curl / HTTP スクリプト** — 起動中の dev サーバに対して。
3. **CLI 実行 + fixture 入力** — stdout を既知の正答スナップショットと diff。
4. **ヘッドレスブラウザスクリプト**（Playwright / Puppeteer）— UI を駆動し、DOM・console・network にアサート。
5. **採取済みトレースの再生** — 実際のリクエスト / ペイロード / イベントログをディスクに保存し、対象コードパスに隔離再生。
6. **使い捨てハーネス** — システムの最小サブセット（1サービス+モック依存）を立ち上げ、関数呼び出し1発でバグのコードパスを動かす。
7. **プロパティ / fuzz ループ** — 「時々出力が壊れる」系なら、ランダム入力1000件を回して失敗モードを探す。
8. **bisect ハーネス** — 2つの既知の状態（コミット・データセット・バージョン）の間で現れたなら、「状態 X で起動 → 確認 → 繰り返し」を自動化して `git bisect run` に乗せる。
9. **差分実行ループ** — 同じ入力を旧版と新版（または2つの設定）に通して出力を diff。
10. **HITL スクリプト** — 最後の手段。人間のクリックが必要なら、[scripts/hitl-loop.template.cjs](scripts/hitl-loop.template.cjs) で人間の操作自体を構造化して駆動する。捕捉した値がエージェントに返る。

正しいループが作れたら、バグは9割直ったも同然。

## ループ自体を改善する

ループはプロダクトとして扱う。1本できたら問う:

- **速くできるか?** セットアップのキャッシュ、無関係な初期化のスキップ、テストスコープの絞り込み。
- **シグナルを鋭くできるか?** 「クラッシュしなかった」ではなく、特定の症状そのものにアサートする。
- **決定的にできるか?** 時刻の固定、乱数のシード、ファイルシステムの隔離、ネットワークの凍結。

30秒かかるフレーキーなループは、ループなしと大差ない。2秒の決定的ループはデバッグの超能力。

## 非決定的なバグ

目標はきれいな再現ではなく**再現率の引き上げ**。トリガーを100回ループ、並列化、負荷追加、タイミング窓の絞り込み、sleep の注入。フレーク率50%ならデバッグできる。1%では無理 — デバッグ可能になるまで率を上げ続ける。
