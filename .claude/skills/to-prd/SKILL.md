# To PRD

出典: [mattpocock/skills](https://github.com/mattpocock/skills) の to-prd を改変（MIT — 表記は [LICENSE](../../../LICENSE)）。

**発動条件**: description のとおり。
**書き込み**: `docs/specs/<機能スラッグ>.md`（1機能1ファイル）のみ。

## 手順

1. まだなら探索する: [docs/PRODUCT.md](../../../docs/PRODUCT.md)（特に「用語」節）・触る領域の [docs/adr/](../../../docs/adr/README.md)・コードベースの現状。spec 全体で「用語」節の語彙を使い、adr に従う。
2. テストの継ぎ目（seam）を素描する: 既存の継ぎ目を優先し、できるだけ高い位置のものを使う。新しい継ぎ目が要るなら、可能な限り高い位置で提案する。**継ぎ目がユーザの期待と合っているか確認を取る。**
3. 下のテンプレートで spec を書き、`docs/specs/` に保存する。

## 書き方の規則

- **ファイルパスや具体的なコード片を書かない**（すぐ陳腐化する）。例外: プロトタイプが散文より正確に決定を表すスニペット（状態機械・reducer・スキーマ・型の形）を生んだ場合のみ、決定を担う部分だけを該当決定の中に引用し、プロトタイプ由来であることを一言添える。
- ユーザーストーリーは「<アクター> として、<機能> が欲しい。<利益> のため」の番号付きリストで、機能の全側面を網羅するまで挙げる。

## テンプレート（spec の節構成はここが唯一の正。内部の相対リンクは docs/specs/ から見たパス）

```markdown
# <機能名>

## 問題と解決
ユーザ視点で、直面している問題と、その解決をそれぞれ簡潔に。

## ユースケース（ユーザーストーリー）
1. <アクター> として、<機能> が欲しい。<利益> のため

## スコープ（作る / 作らない）
この spec でやらないことを明示する。

## データモデル

## 実装決定
変更・新設するモジュールとそのインタフェース、スキーマ変更、API 契約、
アーキテクチャ上の判断、開発者からの技術的クラリフィケーション。

## テスト決定
テスト対象のモジュール / コードベース内の類似テスト（先行事例）。
良いテストの定義は [ARCHITECTURE.md「検証方針」](../ARCHITECTURE.md) に従う。

## 受け入れ条件
- [ ] 条件1（P5 で充足を確認できる粒度で）

## 一方通行決定
この spec が含む一方通行決定への参照（→ [adr/](../adr/README.md)）。なければ「なし」。
```

## 厳守事項

- インタビューしない。不明点が残る場合は spec を書かずに /grill を提案する。
- 既存 spec と同じ機能なら新規作成せず、そのファイルを更新する。
