# TDD

出典: [mattpocock/skills](https://github.com/mattpocock/skills) の tdd を改変（MIT — 表記は [LICENSE](../../../LICENSE)）。

**発動条件**: description のとおり。
**書き込み**: コードとテストのみ。docs への記録は CLAUDE.md の常設ルールと /checkpoint の照合に任せる。

## 原則

テストは公開インタフェース越しに**振る舞い**を検証する。実装は丸ごと変わってよいが、テストは変わらないべき（[ARCHITECTURE.md「コード判断規則」](../../../docs/ARCHITECTURE.md)の同原則の実行手順がこのスキル）。

- 良いテストは統合スタイル: 実際のコードパスを公開 API 越しに動かし、「何ができるか」を仕様のように記述する。内部リファクタを生き延びる。
- 悪いテストは実装に結合: 内部コラボレータのモック、private メソッドのテスト、呼び出し回数の検証。振る舞いが変わっていないのにリファクタで壊れるテストは、実装をテストしていた証拠。

具体例は [tests.md](tests.md)、モックの線引き（システム境界のみ）は [mocking.md](mocking.md)。

## アンチパターン: 水平スライス

**「テストを全部書いてから、実装を全部書く」は禁止。** まとめ書きしたテストは想像上の振る舞いを試験し、データ構造や関数シグネチャという「形」に固着して、本物の変化に鈍感になる。

```text
誤（水平）: RED: test1..test5 → GREEN: impl1..impl5
正（垂直）: RED→GREEN: test1→impl1、test2→impl2、…（1テスト→1実装の繰り返し）
```

各テストは直前のサイクルで学んだことに応答する。書いた直後のコードだから、何の振る舞いが重要でどう検証すべきかが分かっている。

## ワークフロー

### 1. 計画

探索時は [PRODUCT.md](../../../docs/PRODUCT.md)「用語」節の語彙をテスト名・インタフェースに使い、触る領域の [adr/](../../../docs/adr/README.md) に従う。コードを書く前に:

- [ ] 必要なインタフェース変更をユーザに確認する
- [ ] テストする振る舞いを列挙し（実装手順ではなく）、優先順位をユーザに確認する
- [ ] 深いモジュール（小さいインタフェース+深い実装）の機会を探す — 問い: メソッド数を減らせるか / 引数を単純にできるか / 複雑さをもっと内側に隠せるか
- [ ] テスト容易性のためのインタフェース設計（→ [interface-design.md](interface-design.md)）
- [ ] 計画にユーザの承認を得る

全部はテストできない。クリティカルパスと複雑なロジックに集中し、どの振る舞いが最重要かをユーザに確認する。

### 2. トレーサーバレット

システムについて1つのことを確認するテストを1本だけ書く。RED（失敗を見る）→ GREEN（通る最小のコード）。経路が端から端まで通ることの証明になる。

### 3. 漸進ループ

残りの振る舞いごとに RED → GREEN を繰り返す。ルール: 一度に1テスト / 現在のテストを通す分のコードだけ / 将来のテストを先回りしない / 観測可能な振る舞いに集中。

### 4. リファクタ

全テストが通ってから候補を探す。1ステップごとにテストを回す。**RED の間はリファクタしない。** 候補:

- 重複 → 関数 / クラスに抽出
- 長いメソッド → private ヘルパに分割（テストは公開インタフェースのまま）
- 浅いモジュール → 統合するか深くする
- feature envy → ロジックをデータのある場所へ移す
- プリミティブ偏執 → 値オブジェクトを導入
- 新しいコードが問題を露呈させた既存コード

## サイクルごとのチェック

- [ ] テストは実装ではなく振る舞いを記述している
- [ ] 公開インタフェースだけを使っている
- [ ] 内部リファクタを生き延びるテストである
- [ ] コードは現在のテストに対して最小である
- [ ] 投機的な機能を足していない
