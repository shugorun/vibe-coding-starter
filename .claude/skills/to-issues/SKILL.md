# To Issues

出典: [mattpocock/skills](https://github.com/mattpocock/skills) の to-issues を改変（MIT — 表記は [LICENSE](../../../LICENSE)）。

**発動条件**: description のとおり。
**書き込み**: `docs/issues/` への新規ファイル作成のみ（命名・frontmatter・本文の節は [issues/README.md](../../../docs/issues/README.md) が唯一の正）。既存 issue の更新はしない（診断の進行は /diagnose、補完は /checkpoint が担当）。

## (b) 単発の問題の起票

会話に出た問題を「問題」書式で1件1ファイル起票して終わり。再現手順が不明なら、分かっている範囲を書き「未確認」と明記する。

## (a) 計画・spec の分解起票

### 1. 文脈を集める

会話中の計画・spec を入力にする。spec への参照（パス）が渡されたら全文を読む。

### 2. 探索（まだなら）

タイトル・本文は [PRODUCT.md](../../../docs/PRODUCT.md)「用語」節の語彙を使い、触る領域の [adr/](../../../docs/adr/README.md) に従う。

### 3. 垂直スライス（トレーサーバレット）に分解する

各 issue は**全レイヤを端から端まで貫く薄いスライス**であって、1レイヤの水平スライスではない。

- 各スライスは狭くても**完全**な経路（スキーマ・ロジック・UI・テスト）を1本通す
- 完了したスライスは単体でデモまたは検証できる
- 厚い少数より薄い多数を選ぶ

各スライスを HITL / AFK に区分する（区分の定義は [issues/README.md](../../../docs/issues/README.md)）。可能な限り AFK に寄せる。

### 4. ユーザに確認する

分解案を番号付きリストで提示する。各スライスに: タイトル / HITL・AFK / 何にブロックされるか / カバーするユーザーストーリー（元資料にあれば）。

- 粒度は適切か（粗すぎ / 細かすぎ）
- 依存関係は正しいか
- 統合・分割すべきスライスはないか
- HITL / AFK の区分は正しいか

承認が出るまで反復する。

### 5. 起票する

承認されたスライスごとに issues/README.md の「実装スライス」書式でファイルを作成する。**ブロックする側から順に**作成する。番号・命名・書式は README のとおり。

## 厳守事項

- 元になった spec・計画ファイルを変更・クローズしない。
- 起票したファイル名の一覧を最後に報告する。
