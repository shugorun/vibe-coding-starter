# CLAUDE.md

**docs 駆動の個人開発リポジトリ。** バイブコーディングを透明性付きで進める。コードと並行して、判断・要件・仕様・詰まり・次タスクを `docs/` に残す。`docs/` 配下の md を能動的に作成・更新するのは、このプロジェクトで明示的に要求された動作（「勝手に md を作らない」既定はここでは適用しない）。

## フェーズ（順に進む。現在地は `docs/05-PROGRESS.md`）

P0 初期化 → P1 アイデア → P2 MVP → P3 app-design → P4 spec → P5 本実装 → P6 リリース。

- **各フェーズの移行ゲートが全チェックになるまで次へ進まない。** ゲートは「次フェーズに必要な最低限の決断・入力」を含む（定義は `docs/03-ROADMAP.md`）。例: MVP を作る手段（種別・言語）は P1 で決める。
- **MVP(P2) は `mvp/`、本実装(P5) は `app/` と別ディレクトリ。** MVP は使い捨て。本実装は MVP のコード/ロジックも**技術選定も**引き継がず 0 から決め直す（MVP は安く作る手段を選んでよい。本実装の種別・言語・スタックは P3 で 0 から確定）。
- 技術検証のスパイクは `poc/`（使い捨て・どのフェーズでも。`app/` を汚さない。結論は `docs/research/` へ）。

## 作業のしかた: `/work`

自然文の作業依頼（「作って / 実装して / 直して / 詰めて / 調べて」等）には `/work` を実行する。**一直線の手順ではなく、固定点＋ループ**:

- **着手前（固定点）**: 確認 — ブロッカー質問と**ユーザがやるべきタスク**（認証情報・アカウント・課金・外部サービス設定・実機準備・ユーザしか決められない選択）を洗い出す。未解決のまま着手しない。
- **作業ループ**: 依頼を作業単位（= コミット1個分）に分解し、各単位を進める。コードと docs を行き来してよい。**随時発火する不変条件**: 触る前に規則を読む（`08-CODING` / `09-ENVIRONMENT` / 記録ルール）・解決すべき問題に当たったら issue 起票（`docs/issues/`）・非自明な判断をしたら ADR 起票（`docs/decisions/`）。
- **区切り（固定点）**: `/checkpoint`（progress 追記 → 検証緑 → commit。push しない）。次の単位があればループに戻る。

開始は `/catchup-project`、終了は `/wrapup-project`（commit + push まで）。

## 記録の判断基準（量でなく重要度）

記録するのは「将来の自分がこれ無しだと **間違える / 迷う**」情報だけ。

- 記録する: 方向転換、確定した判断、ユーザが明示した好み・制約、再導出できない背景、未確定事項、解決すべき問題（issue）、環境の制約・回避策（09-ENVIRONMENT）。
- 記録しない: やったこと自体（git が持つ）、既存記述の言い換え、まだ揺れている途中の思考。

運用の詳細（記録先の表・cadence・レビューゲート・status・commit prefix）は `docs/02-GUIDELINES.md` が単一の正。ここでは繰り返さない。

## 透明性

非自明なコマンド（git・ビルド・削除・外部呼び出し等）は実行前に1行で何をするか宣言する。バックグラウンド起動は、何が動くか・どう確認/停止するかを明示する。
