# CLAUDE.md

**docs 駆動の個人開発リポジトリ。** バイブコーディングを透明性付きで進める。コードと並行して、判断・要件・仕様・詰まり・次タスクを `docs/` に残す。`docs/` 配下の md を能動的に作成・更新するのは、このプロジェクトで明示的に要求された動作（「勝手に md を作らない」既定はここでは適用しない）。

## フェーズ（順に進む。現在地は `docs/05-PROGRESS.md`）

P0 初期化 → P1 アイデア → P2 MVP → P3 app-design → P4 spec → P5 本実装 → P6 リリース。

- **各フェーズの移行ゲートが全チェックになるまで次へ進まない。** ゲート定義は `docs/03-ROADMAP.md`。
- **MVP(P2) は `mvp/`、本実装(P5) は `app/` と別ディレクトリ。** MVP は使い捨て。本実装は MVP のコード/ロジックを引き継がず 0 から書く。

## 作業のしかた: `/work`

自然文の作業依頼（「作って / 実装して / 直して / 詰めて / 調べて」等）には `/work` を実行する。4ステップを順に行い、各ステップで該当規則を読む:

1. **確認** — 着手前に、ブロッカーになる質問と**ユーザがやるべきタスク**（認証情報・アカウント・課金・外部サービス設定・実機準備・ユーザしか決められない選択）を洗い出す。未解決のまま着手しない。
2. **コーディング** — `docs/08-CODING.md`（規則）と `docs/09-ENVIRONMENT.md`（環境）を読んでから実装。詰まり・罠を踏んだら `docs/issues/` に起票。
3. **docs 更新** — 記録規則（下記）に従い更新。非自明な判断をしたら `docs/decisions/`（ADR）に起票。
4. **commit** — `/checkpoint`（progress 追記 → 検証緑 → commit。push しない）。

開始は `/catchup-project`、終了は `/wrapup-project`（commit + push まで）。

## 記録の判断基準（量でなく重要度）

記録するのは「将来の自分がこれ無しだと **間違える / 迷う**」情報だけ。

- 記録する: 方向転換、確定した判断、ユーザが明示した好み・制約、再導出できない背景、未確定事項、踏んだ罠（issue）、環境の罠（09-ENVIRONMENT）。
- 記録しない: やったこと自体（git が持つ）、既存記述の言い換え、まだ揺れている途中の思考。

運用の詳細（記録先の表・cadence・レビューゲート・status・commit prefix）は `docs/02-GUIDELINES.md` が単一の正。ここでは繰り返さない。

## 透明性

非自明なコマンド（git・ビルド・削除・外部呼び出し等）は実行前に1行で何をするか宣言する。バックグラウンド起動は、何が動くか・どう確認/停止するかを明示する。
